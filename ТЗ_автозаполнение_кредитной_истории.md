# ТЗ: Автозаполнение анкеты из HTML-отчёта кредитной истории (InfoScore/CIAC)

## 1. Контекст и цель

**Приложение:** FastAPI + SQLAlchemy + vanilla JS (SPA). Уже работает на продакшне.

**Цель:** Дать пользователю кнопку «Загрузить из КИ» в разделе **«Кредитная история»** анкеты. Пользователь выбирает HTML-файл отчёта InfoScore (кредитное бюро ЦИКК, Узбекистан), сервер его парсит и возвращает JSON с извлечёнными данными, фронт автоматически заполняет поля анкеты.

### Принцип заполнения полей

Поддерживаются **оба сценария** — загрузка HTML и ручной ввод:

| Поле | Авто из HTML | Фаза 1 (тест) | Фаза 2 (прод) | Примечание |
|---|---|---|---|---|
| `full_name`, `pinfl`, `birth_date` | ✅ | ✅ ручной | ✅ ручной | Личные данные всегда редактируемы |
| `has_current_obligations`, `obligations_count` | ✅ | ✅ ручной | 🔒 readonly | Из HTML в Фазе 2 |
| `total_obligations_amount`, `monthly_obligations_payment` | ✅ | ✅ ручной | 🔒 readonly | Из HTML в Фазе 2 |
| `closed_obligations_count` | ✅ | ✅ ручной | 🔒 readonly | Из HTML в Фазе 2 |
| `max_overdue_*`, `overdue_category`, `last_overdue_date` | ✅ | ✅ ручной | 🔒 readonly | Из HTML в Фазе 2 |
| `overdue_reason` | ❌ | ✅ ручной | ✅ ручной | Комментарий андеррайтера — всегда вручную |
| `ki_score` (Балл КИ) | ✅ | 🔒 readonly | 🔒 readonly | Только из файла в обеих фазах |

**Логика UX — двухфазный ввод:**

**Фаза 1 (тестирование, текущий деплой):**
- Загрузка HTML → поля заполняются автоматически, затем **остаются редактируемыми**
- Нет HTML-файла → ручной ввод как сейчас
- `ki_score` — всегда `readonly` в обеих фазах

**Фаза 2 (после успешного прохождения тестов):**
- Поля `has_current_obligations`, `obligations_count`, `total_obligations_amount`, `monthly_obligations_payment`, `closed_obligations_count`, `max_overdue_principal_days`, `max_overdue_principal_amount`, `max_continuous_overdue_percent_days`, `max_overdue_percent_amount`, `overdue_category`, `last_overdue_date` переводятся в **`readonly`** — заполнение только через загрузку HTML
- Кнопка загрузки HTML становится **обязательным шагом** перед сохранением анкеты
- `overdue_reason` остаётся редактируемым всегда (это комментарий андеррайтера)

**Реализация переключения фаз:** добавить булевый флаг `KI_AUTOFILL_STRICT = False` в конфиг (env-переменная). При `True` — поля readonly, при `False` — редактируемые. Opus переключает флаг после подтверждения тестов.

## 2. Структура HTML-отчёта InfoScore (что извлекать)

Отчёт имеет секции. Все тексты в отчёте — на узбекском языке (латиница). Стандартный формат: `177 "InfoScore"`.

### Секция 1 — KREDIT AXBOROTI SUBYEKTI (личные данные)

HTML-структура: `<section>` → внутри сначала идут **все лейблы**, затем **все значения** (не строка-за-строкой, а двумя блоками).

| Лейбл в HTML | Поле анкеты | Примечание |
|---|---|---|
| `F.I.O.:` | `full_name` | Следующее за блоком лейблов значение |
| `JShShIR:` | `pinfl` | Убрать пробелы → 14 цифр |
| `Tug'ilgan sana:` | `birth_date` | Формат `YYYY-MM-DD` |

**Важно:** Лейблы и значения не чередуются построчно. В секции сначала идут все лейблы подряд (F.I.O., JShShIR, Tug'ilgan sana, Jinsi, Huquqiy maqomi, ...), затем все значения подряд. Порядок значений соответствует порядку лейблов. Нужно подсчитать количество лейблов и взять значения начиная с offset.

**Правильный способ парсинга секции 1:**
```python
section = soup.find(string=lambda t: t and 'KREDIT AXBOROTI SUBYEKTI' in t)
container = section.find_parent('section')  # или родительский блок
labels = ['F.I.O.:', 'JShShIR:', "Tug'ilgan sana:", 'Jinsi:',
          'Huquqiy maqomi:', "Ro'yhatdan o'tgan manzili:",
          'Yashash manzili:', 'Telefon raqami:', 'Elektron pochtasi:']
# Найти индекс последнего лейбла в texts, затем взять значения по offset
```

### Секция 4 — UMUMLASHTIRILGAN MA'LUMOTLAR (сводная статистика)

Структура: `значение — '-' — описание`. То есть сначала идёт число, потом строка `'-'`, потом текст-описание.

| Описание в HTML | Поле анкеты | Тип |
|---|---|---|
| `asosiy qarz (AQ) bo'yicha muddati o'tgan to'lovlar soni` | *(только для внутренней логики)* | int |
| `muddati o'tgan AQ maksimal kuni` | `max_overdue_principal_days` | int |
| `muddati o'tgan AQ maksimal summasi` | `max_overdue_principal_amount` | float |
| `uzluksiz muddati o'tgan foiz to'lovlarining maksimal kuni` | `max_continuous_overdue_percent_days` | int |
| `muddati o'tgan foiz to'lovlarining maksimal summasi` | `max_overdue_percent_amount` | float |

**Правильный способ парсинга:** Для каждого описания найти его индекс в `texts`, затем взять `texts[i-2]` (через `'-'`):
```
texts[i-2] = "23"      ← значение
texts[i-1] = "-"
texts[i]   = "muddati o'tgan AQ maksimal kuni"
```

### Секция 5 — AMALDAGI SHARTNOMALAR (активные договоры)

Таблица с колонками: `№ | QARZ BERUVCHI | SHARTNOMA RAQAMI | VALYUTA | UMUMIY QARZ QOLDIG'I | MUDDATI O'TGAN QISMI | O'RTACHA OYLIK TO'LOV`

Строка `Jami` (итог) — последняя строка таблицы. За ней идут:
- `texts[jami_idx + 1]` → общий остаток долга → `total_obligations_amount`
- `texts[jami_idx + 2]` → общая просроченная сумма *(не используется)*
- `texts[jami_idx + 3]` → общий средний ежемесячный платёж → `monthly_obligations_payment`

**Количество активных договоров:** Подсчитать строки таблицы между заголовком и `Jami`. Строки нумерованы: `1`, `2`, `3`, ... Найти максимальный порядковый номер.

| Поле анкеты | Источник |
|---|---|
| `obligations_count` | Количество строк в таблице (до Jami) |
| `total_obligations_amount` | `Jami → texts[+1]`, убрать пробелы, float |
| `monthly_obligations_payment` | `Jami → texts[+3]`, убрать пробелы, float |
| `has_current_obligations` | `"есть"` если count > 0, иначе `"нет"` |

---

### Количество закрытых договоров

Слово `Yopiq` (узб. «закрыт») встречается один раз на каждый закрытый договор в разделе истории. Просто посчитать вхождения в `texts`.

| Поле анкеты | Источник |
|---|---|
| `closed_obligations_count` | `texts.count('Yopiq')` |

---

### Определение категории просрочки

На основе `max_overdue_principal_days`:

| Значение | `overdue_category` |
|---|---|
| 0 и `_overdue_count == 0` | `"до 30 дней"` (чистая история) |
| 1–30 | `"до 30 дней"` |
| 31–60 | `"31-60"` |
| 61–90 | `"61-90"` |
| > 90 | `"90+"` |

---

### Дата последней просрочки

В детальных блоках договоров просроченные платежи сопровождаются датой в формате `[YYYY-MM-DD]`. Нужно найти все такие даты рядом со строками `"Asosiy qarz bo'yicha muddati o'tgan to'lovlar"` или `"Foiz bo'yicha muddati o'tgan to'lovlar"` и взять **максимальную** (самую позднюю).

| Поле анкеты | Источник |
|---|---|
| `last_overdue_date` | `max()` дат рядом с маркерами просрочки |

### Балл КИ (новое поле — не путать с internal `risk_grade`)

InfoScore содержит два значения скоринга:
- **Числовой балл** (`SKORING BALL:` / `Скоринговый балл:`): например `368`, `302`
- **Класс оценки** (`BAHOLASH SINFI:` / `Класс оценки:`): например `"B3, Yaxshi daraja"` → код `B3`

Эти данные **не должны** попадать в существующее поле `risk_grade`, которое является **внутренним** скорингом приложения (назначается андеррайтером/системой независимо от InfoScore).

Создать новое поле `ki_score` (строка) — хранит составное значение: `"B3 / 368"`.

| Новое поле БД | Источник в HTML | Формат хранения |
|---|---|---|
| `ki_score` | Класс + числовой балл (секция 2) | `"B3 / 368"` или просто `"B3"` если балл отсутствует |

```

birth_date                        ← Tug'ilgan sana (секция 1)

monthly_obligations_payment       ← Jami[+3] (секция 5)

max_overdue_principal_days        ← секция 4

max_overdue_percent_amount        ← секция 4

```

**Не заполнять автоматически:** `overdue_reason`, `overdue_check_result` (авто), `overdue_reason`.

### `app/routers/anketa.py` — добавить эндпоинт

**Метод:** `POST`
**URL:** `/api/anketas/{anketa_id}/parse-credit-report`
**Auth:** требуется (`get_current_user`)
**Content-Type:** `multipart/form-data`
**Параметры:** `file: UploadFile` (HTML-файл отчёта КИ)

**Логика:**
1. Проверить что анкета существует и пользователь имеет к ней доступ (`check_anketa_access`)
2. Прочитать содержимое файла: `content = await file.read()`
3. Декодировать: `html = content.decode('utf-8')`
4. Вызвать: `parsed = parse_infoscore_html(html)` из `app.credit_report_parser`
5. Вернуть JSON: `{"parsed": parsed}`

**Важно:** Эндпоинт только **парсит и возвращает** данные — он **не сохраняет** их в базу. Сохранение делает уже существующий `PATCH /api/anketas/{anketa_id}` — вызов происходит на стороне JS после подтверждения.

**Импорты добавить в anketa.py:**
```python
from fastapi import UploadFile, File
from app.credit_report_parser import parse_infoscore_html
```

**Код эндпоинта:**
```python
@router.post("/{anketa_id}/parse-credit-report")
async def parse_credit_report(
    anketa_id: int,
    file: UploadFile = File(...),
    user: User = Depends(get_current_user),
    db: Session = Depends(get_db),
):
    anketa = db.query(Anketa).filter(Anketa.id == anketa_id).first()
    if not anketa:
        raise HTTPException(status_code=404, detail="Анкета не найдена")
    check_anketa_access(anketa, user, db)

    content = await file.read()
    try:
        html = content.decode("utf-8")
    except UnicodeDecodeError:
        html = content.decode("cp1251", errors="replace")

    parsed = parse_infoscore_html(html)
    return {"parsed": parsed}
```

### `app/static/js/app.js` — добавить функцию

Добавить функцию `handleKiUpload` (можно в конец файла):

```javascript
async function handleKiUpload(event) {
  const file = event.target.files[0];
  if (!file) return;

  const anketaId = currentAnketaId; // глобальная переменная уже используется в приложении
  if (!anketaId) return;

  const statusEl = document.getElementById('kiUploadStatus');
  statusEl.style.display = 'block';
  statusEl.innerHTML = '<span style="color:var(--text-light)">Анализирую отчёт...</span>';

  const formData = new FormData();
  formData.append('file', file);

  try {
    const token = localStorage.getItem('token');
    const resp = await fetch(`/api/anketas/${anketaId}/parse-credit-report`, {
      method: 'POST',
      headers: { 'Authorization': 'Bearer ' + token },
      body: formData,
    });

    if (!resp.ok) {
      const err = await resp.json();
      statusEl.innerHTML = `<span style="color:var(--red)">Ошибка: ${err.detail || 'Не удалось прочитать файл'}</span>`;
      return;
    }

    const data = await resp.json();
    const parsed = data.parsed;

    // ── Заполнить поля формы ──
    const fieldMap = {
      full_name: 'f-full_name',
      pinfl: 'f-pinfl',
      birth_date: 'f-birth_date',
      has_current_obligations: 'f-has_current_obligations',
      obligations_count: 'f-obligations_count',
      total_obligations_amount: 'f-total_obligations_amount',
      monthly_obligations_payment: 'f-monthly_obligations_payment',
      closed_obligations_count: 'f-closed_obligations_count',
      max_overdue_principal_days: 'f-max_overdue_principal_days',
      max_overdue_principal_amount: 'f-max_overdue_principal_amount',
      max_continuous_overdue_percent_days: 'f-max_continuous_overdue_percent_days',
      max_overdue_percent_amount: 'f-max_overdue_percent_amount',
      overdue_category: 'f-overdue_category',
      last_overdue_date: 'f-last_overdue_date',
      ki_score: 'f-ki_score',   // новое поле — НЕ 'f-risk_grade'
    };

    let filled = 0;
    for (const [field, elId] of Object.entries(fieldMap)) {
      if (parsed[field] !== undefined && parsed[field] !== null) {
        const el = document.getElementById(elId);
        if (el) {
          el.value = parsed[field];
          el.dispatchEvent(new Event('input'));   // обновить DTI / live-calc
          el.dispatchEvent(new Event('change'));
          filled++;
        }
      }
    }

    statusEl.innerHTML = `<span style="color:var(--green)">✓ Заполнено ${filled} полей из отчёта. Проверьте данные перед сохранением.</span>`;

    // Запустить пересчёт DTI
    if (typeof runClientCalc === 'function') runClientCalc();

    // Сбросить input чтобы можно было загрузить тот же файл ещё раз
    event.target.value = '';

  } catch (e) {
    statusEl.innerHTML = `<span style="color:var(--red)">Ошибка соединения: ${e.message}</span>`;
  }
}
```

## 9. Поддержка русскоязычных отчётов

InfoScore выдаёт отчёты на двух языках — узбекском (латиница) и русском. Структура HTML одинакова, меняются только текстовые лейблы. Парсер обязан поддерживать оба варианта.

### Определение языка отчёта

```python
def detect_language(texts: list[str]) -> str:
    """Возвращает 'uz' или 'ru'."""
    if any('СУБЪЕКТ КРЕДИТНОЙ' in t for t in texts):
        return 'ru'
    return 'uz'  # по умолчанию узбекский
```

### Таблица лейблов: Узбекский ↔ Русский

#### Секция 1 — Личные данные

| Поле анкеты | Узбекский лейбл | Русский лейбл |
|---|---|---|
| `full_name` | `F.I.O.:` | `Наименование:` |
| `pinfl` | `JShShIR:` | `ПИНФЛ:` |
| `birth_date` | `Tug'ilgan sana:` | `Дата рождения:` |

#### Секция 2 — Скоринг (новое поле `ki_score`)

| Новое поле БД | Узбекский лейбл | Русский лейбл |
|---|---|---|
| `ki_score` (числовой балл) | `SKORING BALL:` | `Скоринговый балл:` |
| `ki_score` (класс) | `BAHOLASH SINFI:` | `Класс оценки:` |

Извлечь оба значения и объединить: `"B3 / 368"`. Класс — regex `[A-Z]\d+` из строки вида `"B3, Yaxshi daraja"`. Балл — следующее число после лейбла `SKORING BALL:` / `Скоринговый балл:`.

⚠️ Результат записывается в **новое поле `ki_score`**, а не в существующий `risk_grade`.

#### Секция 4 — Сводная статистика

| Поле анкеты | Узбекский лейбл | Русский лейбл |
|---|---|---|
| *(overdue count, internal)* | `asosiy qarz (AQ) bo'yicha muddati o'tgan to'lovlar soni` | `количество просрочек основного долга (ОД)` |
| `max_overdue_principal_days` | `muddati o'tgan AQ maksimal kuni` | `максимальная просрочка ОД (дни)` |
| `max_overdue_principal_amount` | `muddati o'tgan AQ maksimal summasi` | `максимальная просрочка ОД (сумма)` |
| `max_continuous_overdue_percent_days` | `uzluksiz muddati o'tgan foiz to'lovlarining maksimal kuni` | `максимальная непрерывная просрочка % (дни)` |
| `max_overdue_percent_amount` | `muddati o'tgan foiz to'lovlarining maksimal summasi` | `максимальная просрочка % (сумма)` |

Паттерн парсинга одинаков для обоих языков: `texts[i-2]` → значение, `texts[i-1]` → `'-'`, `texts[i]` → лейбл.

#### Секция 5 — Активные договора

| | Узбекский | Русский |
|---|---|---|
| Заголовок секции | `AMALDAGI SHARTNOMALAR` | `ДЕЙСТВУЮЩИЕ ДОГОВОРА` |
| Строка итогов | `Jami` | `Итого` |
| Нет договоров | *(отсутствует)* | `Не имеется` |

**Важно:** Если за заголовком секции сразу следует `"Не имеется"` — активных обязательств нет:
```python
has_current_obligations = "нет"
obligations_count = 0
total_obligations_amount = None
monthly_obligations_payment = None
```

#### Закрытые договора

| | Узбекский | Русский |
|---|---|---|
| Маркер закрытого договора | `Yopiq` | `Закрыт` |

Подсчёт: `texts.count('Yopiq') + texts.count('Закрыт')` — на случай если в одном файле оба языка.

## 10. Поддержка юридических лиц (юрик)

### Определение типа субъекта

Парсер автоматически определяет — физик или юрик — и возвращает `entity_type` в ответе.

**Проблема:** в русской версии оба типа используют одинаковый лейбл `Наименование:` для имени. Единственные надёжные маркеры для разграничения:

| Маркер | Физик | Юрик |
|---|---|---|
| Uzbek: лейбл ID | `JShShIR:` | `STIRi:` |
| Russian: лейбл ID | `ПИНФЛ:` | `ИНН:` |
| Uzbek: статус | `Jismoniy shaxs` | `Yuridik shaxs` |
| Russian: статус | `Физическое лицо` | `Юридическое лицо` |

Приоритетный способ — проверить лейбл ID (он всегда присутствует и уникален):

```python
def detect_entity_type(texts: list[str]) -> str:
    """Возвращает 'individual' или 'legal_entity'."""
    # Первичный маркер: лейбл ID (самый надёжный)
    if 'STIRi:' in texts or 'ИНН:' in texts:
        return 'legal_entity'
    if 'JShShIR:' in texts or 'ПИНФЛ:' in texts:
        return 'individual'
    # Запасной маркер: статус субъекта
    if 'Yuridik shaxs' in texts or 'Юридическое лицо' in texts:
        return 'legal_entity'
    return 'individual'  # по умолчанию
```

Ответ эндпоинта дополняется полем: `{"entity_type": "legal_entity", "parsed": {...}}`.

## 11. Ограничения и Edge Cases

- Поддерживается только формат **InfoScore 177** (ЦИКК, Узбекистан). Другие форматы не парсить, возвращать ошибку `{"detail": "Неподдерживаемый формат отчёта"}`.
- Если отчёт содержит несколько страниц (дублирование HTML), данные не задваиваются — парсер берёт первое вхождение.
- Валюта: все суммы в **UZS** (сум). Конвертация не требуется.
- Если `pinfl` не 14 цифр — не включать в результат.
- Поля из анкеты, которых нет в отчёте (`overdue_reason`, `partner` и пр.), не включать в ответ — пользователь заполняет вручную.
- Кнопка загрузки видна **только в режиме редактирования черновика** (status === 'draft'). Если статус иной — скрыть блок `kiUploadBlock`.
- **Env-флаг `KI_AUTOFILL_STRICT`** (по умолчанию `False`): при `True` поля КИ становятся `readonly` на фронте и загрузка HTML становится обязательной. Передаётся на фронт через существующий эндпоинт `/api/health` или отдельный `/api/config`. В Фазе 1 флаг `False`. После успешного тестирования Opus выставляет `True` в `.env`.
